## TL;DR - Generate Your Resilience Log in 5 Steps

### Step 1: Clean Old Logs


### Step 3: Connect MiniFix & Send Orders

```powershell

```

```powershell

# Optional: Send 1 more order to trigger ResendRequest

# Terminal 2: Copy the resilience log

Copy-Item "FIX.4.4-EXEC_SERVER-MINIFIX_CLIENT.session" "LAB10_RESILIENCE_LOG.txt"
```

## What to Look For in the Log

The log file will contain QuickFIX session events. Look for these key patterns:

### ✅ Session Created
```
Session FIX.4.4:EXEC_SERVER->MINIFIX_CLIENT created
```

### ✅ Messages Processed (Before Crash)
```
Message processed: MsgSeqNum=2 MsgType=D
Message processed: MsgSeqNum=3 MsgType=D
Message processed: MsgSeqNum=4 MsgType=D
...
```

### ✅ Disconnect Event
```
Disconnect detected
```

### ✅ Reconnection
```
Session reconnected
Logon received
```

### ⭐ ResendRequest (The Key Evidence!)
```
MsgSeqNum too high, expecting X but received Y
Sending ResendRequest FROM=X TO=0
```

### ✅ Message Recovery (PossDup=Y)
```
Resend of MsgSeqNum=X received (PossDup=Y)
```

## What If ResendRequest Doesn't Appear?

ResendRequest only appears if you have a **real sequence gap**. Best practices:

### Option A: Kill Java Process (Most Reliable)
```powershell
# This guarantees sequence loss → ResendRequest
Get-Process java | Stop-Process -Force
```

### Option B: Network Disconnect Via MiniFix
1. In MiniFix, click **Disconnect**
2. Wait 3 seconds
3. Send an order from MiniFix (it buffers locally)
4. Click **Connect** back to Java
5. The buffered order may arrive out of sequence

### Option C: Force Database Failure
```bash
# Stop PostgreSQL (triggers retry + potential gap)
sudo systemctl stop postgresql

# Send more orders (they queue while DB is down)
# Restart PostgreSQL
sudo systemctl start postgresql
```

## Most Common Issues

| Problem | Solution |
|---------|----------|
| No log file created | Ensure Order Service ran to completion; check `logs/sessions/` directory exists |
| Log file is empty | Session didn't process messages; run Order Service longer and send more orders |
| Can't find ResendRequest | That's OK! Just having the continuous sequence (no reset) is valid for this test |
| Java process won't stop | Try `Get-Process java \| Stop-Process -Force` in PowerShell |
| Permission denied copying | Close MiniFix and Order Service first; try `Copy-Item` without `-Force` |

## FAQ

**Q: Do I NEED to see ResendRequest?**  
A: No. The key resilience evidence is: "Sequence numbers preserved from file store" despite restart. ResendRequest is just bonus.

**Q: What if MiniFix doesn't reconnect automatically?**  
A: Click "Connect" button manually in MiniFix after Order Service restarts.

**Q: Can I use the sample log instead?**  
A: The sample is for reference. Your submission should include your own actual log.

**Q: How long should the test run?**  
A: 30-60 seconds is typical (10-15 second baseline + 5 sec crash + 10 sec recovery + 5 sec verification)

**Q: Where do I submit the log?**  
A: Email or upload: `stocker/cmt/LAB10_RESILIENCE_LOG.txt`
